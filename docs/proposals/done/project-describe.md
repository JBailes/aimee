# Proposal: Auto-Detect and Describe Projects

## Problem

When agents work across workspace projects, they have no structural context — no knowledge of what language a project uses, how to build/test it, what its entry points are, or what it does. The workspace yaml supports `description`, `language`, `build`, `test`, and `lint` fields per project, but they're currently empty for every project. Agents waste turns grepping and exploring to discover what aimee could have told them upfront.

This must be automatic. Every project added to the workspace should get a comprehensive structural description generated by aimee, stored persistently, and injected into every agent's context.

## Approach

`aimee describe` delegates an agent to analyze each project, understand its structure deeply, and produce a structured description that eliminates the need for future agents to explore. Aimee is the intelligence — it reads the code, understands the architecture, and writes the map once so no agent has to rediscover it.

### What Gets Generated

For each project, aimee produces a **project description file** stored at `~/.config/aimee/projects/<name>.md`. This is a structured document covering:

1. **Summary** — what the project is, what it does, in 2-3 sentences
2. **Type** — cli, server, library, webapp, game, infrastructure, etc.
3. **Language / Stack** — primary language(s), frameworks, key dependencies
4. **Build / Test / Lint** — exact commands that work for this project
5. **Entry Points** — where execution starts (e.g. `src/main.c:main()`, `index.ts`)
6. **Directory Layout** — what each top-level directory contains and its purpose
7. **Key Files** — the important files an agent would need to know about, with a one-line role for each
8. **Architecture** — how the major modules connect, what calls what, data flow
9. **Conventions** — naming patterns, code style, patterns used in this project

This is not a README. It's an agent-oriented structural map — the information an agent needs to navigate the project without searching.

### How It Works

**`aimee describe [project-name]`**

1. Load the workspace manifest to get the project list and root paths
2. For each project to describe (all if no name given, or the named one):
   a. Delegate an agent with tools (`--tools`) into the project directory (`--context-dir`)
   b. The delegate's prompt instructs it to:
      - List the directory structure
      - Read key files (entry points, config, build files, READMEs)
      - Identify the architecture, modules, and how they connect
      - Produce the structured description in the defined format
   c. Parse the agent's response and write it to `~/.config/aimee/projects/<name>.md`
3. Update workspace yaml with detected `language`, `build`, `test`, `lint` fields
4. Save the workspace manifest

**Flags:**
- No args: describe all projects that don't yet have a description file
- `--force`: re-describe even if a description file already exists
- `--project <name>`: describe a specific project

**Auto-trigger:** After `workspace_provision` clones a new repo, automatically run describe for that project.

### Context Injection

`workspace_build_context()` is updated to read and include project descriptions:

1. For each project, check if `~/.config/aimee/projects/<name>.md` exists
2. If so, include its contents in the context output
3. If not, fall back to the existing minimal format (name + description field from yaml)

This means every agent — delegates, chat sessions, webchat — gets the full structural map in its system prompt. No grepping needed.

### Example Output

For the `aimee` project, the generated description might look like:

```markdown
# aimee

AI coding assistant with memory, indexing, agent delegation, and web interface.

Type: cli, server
Language: C
Build: cd src && make
Test: cd src && make test
Lint: cd src && make lint

## Entry Points
- src/main.c:main() — CLI entry, dispatches to command handlers
- src/webchat.c:webchat_serve() — HTTPS web server for chat and dashboards

## Directory Layout
- src/ — all C source files and headers
- src/headers/ — header files
- src/tests/ — unit test source files
- src/migrations/ — SQLite schema migrations (SQL files)
- docs/proposals/ — design proposals (complete/ for implemented ones)

## Key Files
- src/main.c — CLI entry point, command dispatch table
- src/agent.c — agent execution loop (tool use, turn management)
- src/agent_context.c — system prompt assembly for delegates
- src/workspace.c — workspace yaml loading, provisioning, context building
- src/webchat.c — HTTPS server, chat UI, dashboard, session management
- src/cmd_core.c — core CLI commands (chat, delegate, webchat, env, describe)
- src/cmd_hooks.c — session hooks, context assembly for external agents
- src/index.c — code indexer (symbols, files, projects)
- src/memory.c — memory system (L1/L2 facts, context assembly)
- src/dashboard.c — web dashboard data endpoints
- src/aimee.h — main header, command table, config structures

## Architecture
CLI dispatches commands through a table in aimee.h. The agent system
(agent.c) runs a tool-use loop: send prompt to LLM, parse tool calls,
execute locally, repeat. Context is assembled per-turn by agent_context.c
which pulls from the SQLite database (memories, index, rules).

The webchat server (webchat.c) is a built-in HTTPS server that serves
a chat interface and dashboard. It uses the same agent execution path
as CLI delegates.

Workspace management (workspace.c) handles multi-project setups:
loading manifests, provisioning dependencies, cloning repos, and
building context strings for agent prompts.
```

### Delegate Prompt

The agent delegated to describe a project receives a prompt like:

```
Analyze the project at {project_path} and produce a structured description.

Read the directory structure, key files, build configuration, and source code
to understand:
- What this project is and does
- What language/stack it uses
- How to build, test, and lint it
- Where execution starts (entry points)
- What each directory contains
- Which files are most important and why
- How the major components connect

Output the description in this exact format:
[format template]

Be thorough but concise. Focus on information that would help an agent
navigate and work in this project without needing to search or explore.
Do not include speculative or aspirational content — only describe what
exists in the code today.
```

### Storage

Descriptions are stored as files rather than in the database because:
- They're large (multi-paragraph structured text)
- They need to be human-readable and editable
- They're per-project, not per-session or per-fact
- They can be version-controlled or shared independently

Location: `~/.config/aimee/projects/<project-name>.md`

### Regeneration on Change

Project descriptions must stay current with the code. Rather than time-based staleness, aimee regenerates descriptions when changes actually land.

**Trigger: `aimee index`**

The code indexer already runs after commits and pulls to update the symbol index. This is the natural place to detect whether a project's structure has changed and needs re-description.

After indexing a project, compare the current state against what the description was built from:
1. Store the commit hash (HEAD at describe time) in the description file's frontmatter
2. After indexing, compare stored hash against current HEAD
3. If they differ, check whether the change is structural:
   - New or deleted files (not just content changes within existing files)
   - Changes to build files (Makefile, package.json, Cargo.toml, etc.)
   - Changes to entry points or top-level directory structure
4. If structural changes detected, queue a re-describe

This avoids re-describing on every typo fix while catching meaningful changes like new modules, restructured directories, or build system changes.

**Description file frontmatter:**

```yaml
---
project: aimee
generated: 2026-03-31T14:22:00Z
commit: 9f697fd
---
```

**Structural change detection** — compare the file list and build files between the stored commit and HEAD:

```
git diff --name-only <stored-commit> HEAD -- . | grep -E '(^[^/]+/$|Makefile|package.json|Cargo.toml|go.mod|\.csproj|CMakeLists)'
```

If this produces output, the project structure may have changed and a re-describe is warranted. If only source file contents changed (edits within existing files), skip it.

**Manual override:** `aimee describe --force` still regenerates unconditionally.

| `workspace.h` | Declare any new public functions |
| `aimee.h` | Register `cmd_describe` in command table |

## Non-Goals

- Not a documentation generator. This produces agent-oriented structural context, not human docs.
- No manual maintenance. The description is generated by aimee's own agent and refreshed on demand.
- Does not replace the code index. The index provides symbol-level search; descriptions provide architectural understanding.
