# Proposal: Comment / AI Slop Checker

## Problem

AI agents produce characteristic comment patterns in generated code: redundant docstrings that restate the function name, "// TODO: implement" left behind, overly verbose inline comments explaining obvious operations, and placeholder comments like "// This function does X". These patterns degrade code quality and are a telltale sign of AI-generated code that hasn't been reviewed.

Evidence: oh-my-openagent implements a comment checker (`src/hooks/comment-checker/`) that intercepts Write/Edit tool calls and scans the diff for known AI slop patterns before the edit is applied. It uses a dedicated CLI tool for pattern detection.

## Goals

- Detect common AI slop patterns in code changes before they are committed
- Flag violations without blocking — append a warning to the tool output
- Cover the most common patterns: redundant docstrings, placeholder TODOs, obvious-comment antipatterns
- Extensible pattern list via configuration

## Approach

Add a post-edit check in the guardrails layer that scans the new content (or diff) for known AI slop patterns. When patterns are detected, append a warning to the tool output suggesting the agent clean up the comments.

### Patterns to detect

1. **Redundant docstrings**: `// Initialize the initializer`, `# Set the settings`
2. **Placeholder TODOs**: `// TODO: implement`, `// TODO: add error handling`
3. **Obvious comments**: `i++ // increment i`, `return result // return the result`
4. **AI attribution**: `// Generated by`, `// Created with AI`
5. **Filler phrases**: `// This function`, `// This method`, `// The following code`

### Changes

| File | Change |
|------|--------|
| `src/guardrails.c` | Add `guardrails_check_comments()` scanning diff content for slop patterns |
| `src/git_verify.c` | Call comment check as optional verification step |
| `src/headers/guardrails.h` | Pattern list and config struct |

## Acceptance Criteria

- [ ] At least 5 common AI slop patterns are detected
- [ ] Detection runs on Edit/Write tool output, not on pre-existing file content
- [ ] Warnings are appended to tool output, not blocking
- [ ] Pattern list is extensible via `.aimee/project.yaml`
- [ ] No false positives on legitimate TODO comments with specific descriptions

## Owner and Effort

- **Owner:** aimee
- **Effort:** M (1–2 days)
- **Dependencies:** None

## Rollout and Rollback

- **Rollout:** Compile-time inclusion; patterns configurable in project.yaml
- **Rollback:** Remove the check; edits proceed without slop scanning
- **Blast radius:** Only affects tool output text for Write/Edit; no behavioral change

## Test Plan

- [ ] Unit test: each pattern type is detected
- [ ] Unit test: legitimate comments (specific TODOs, complex logic explanations) pass
- [ ] Unit test: patterns in string literals are not flagged
- [ ] Integration test: delegate produces slop, verify warning appears

## Operational Impact

- **Metrics:** Count of slop warnings per session
- **Logging:** Log at debug level when patterns are detected
- **Disk/CPU/Memory:** Negligible — regex matching on diff strings

## Priority

| Item | Priority | Effort | Impact |
|------|----------|--------|--------|
| Comment Slop Checker | P1 | M | Medium — improves code quality of agent output |

## Trade-offs

Alternative: block the edit entirely until slop is removed. Too aggressive — some flagged patterns may be intentional. Warning-only approach lets the agent self-correct.

Alternative: run as a git pre-commit hook instead of in the tool pipeline. Misses the opportunity to correct in-flight; the agent has already moved on by commit time.
